## What the Git Reset Removed (code that was written today then reverted)

During the binary search for the regression, we restored files to committed state and selectively re-added changes. Code that was **written today but reverted** because it caused the regression:

### `detectTurnstileViaCDP` — Clean Page WS Upgrade (REVERTED)

The attempted upgrade replaced the simple `Target.getTargets` detection with:

```typescript
// REVERTED — this caused rechallenges
const cleanWs = await this.openCleanPageWs(pageTargetId);
const doc = await cleanWs.send('DOM.getDocument', { depth: -1, pierce: true });
// Walk tree for CF iframe...
const cfType = await cleanWs.send('Runtime.evaluate', {
  expression: `window._cf_chl_opt?.cType || null`
});
cleanWs.close();
```

This was intended to get richer CF type info (`managed`, `non_interactive` etc.) during detection. **Root cause of regression**: even on a fresh `/devtools/page/` WS, `Runtime.evaluate` during the 20-poll detection loop triggers CF's WASM. `DOM.getDocument` alone caused timeouts (never solved). Only `Target.getTargets` is safe for detection.

### `solveTurnstile` — Bail-After-2 + isRechallenge Skip (PARTIALLY REVERTED)

The original refactored version had:
- `bailAfterNoCheckbox = 2` — stop clicking after 2 failures (reduced from 6)
- `isRechallenge` check — skip click attempts on rechallenge, go straight to auto-solve wait

The bail-after-2 and isRechallenge were removed during git reset. **Token polling was re-added** at the end as Phase 2 of the simpler 6-click approach.

### `solveByClicking` — Auto-Solve Wait Loop (REVERTED)

Had a while loop after click attempts that polled `active.aborted` + `state.isSolved()`. This was removed because `isSolved()` calls `Runtime.evaluate` which risks V8 poisoning. The auto-solve path now relies on the existing beacon/navigation handlers.

### `enterSolveWindow` / `exitSolveWindow` (NEVER RE-ADDED)

Coordination methods on the detector that were part of the original day's plan but never survived the binary search. They were meant to prevent navigation-based resolution from racing with the solve loop.

## Binary Search Results (The Key Finding)

| Change | Tested | Result |
|--------|--------|--------|
| Committed code (baseline) | 5/5 | **PASS** |
| + Rechallenge tracking | 5/5 | **PASS** — pure observability |
| + SolveOutcome type | 5/5 | **PASS** — TypeScript refactoring |
| + detectTurnstileViaCDP upgrade (DOM.getDocument + Runtime.evaluate) | 2/5 | **RECHALLENGE on run 2** |
| Fix: remove Runtime.evaluate, keep DOM.getDocument | 1/1 | **TIMEOUT** — 19 events, never solved |
| Fix: revert to Target.getTargets | 5/5 | **PASS** |
| + Token polling in solveTurnstile | 5/5 | **PASS** |
| Ahrefs etsy.com | 1/1 | **PASS** — DR 94 |

**Root cause**: `detectTurnstileViaCDP` runs in a polling loop (20 × 200ms). Any page-level CDP command during that window — even on a fresh clean-page WS — is detected by CF. Only browser-level `Target.getTargets` is safe for detection polling.

**Key refinement from today**: `DOM.getDocument` is safe as a **single call** (Phase 1 clicking) but **unsafe in a polling loop** (detection). The frequency of page interaction matters, not just the command type.
